## v1.17.1 (2026-04-01)

### Bug修复 (PATCH) - 润色优化与评分逻辑改进

#### 1. 修复润色优化提示文案

- 去除"AI检测"相关描述，改为"正在进行润色优化..."
- 简化润色流程，直接进行润色优化

#### 2. 修复重新润色逻辑

**问题描述：**
- 重新润色时基于上一流程重新处理，导致分数可能降低
- 没有基于当前润色后的内容和优化建议进行针对性优化

**解决方案：**
- 重新润色时传递之前的分数、问题和优化建议
- 后端根据历史信息生成针对性优化提示
- 确保每次润色都能针对具体问题改进

#### 3. 修复AI味评分逻辑

**问题描述：**
- 评分按问题类型数量扣分，而非按出现次数扣分
- 导致检测到多个问题但分数仍然很高（如71分显示"低AI味"）
- 评分阈值不合理（70分以上就算"低AI味"）

**解决方案：**
- 改为按问题出现次数扣分：
  - 高严重性问题：每次扣8分
  - 中严重性问题：每次扣4分
  - 低严重性问题：每次扣2分
- 调整评分阈值：
  - 80分以上：低AI味（好）
  - 50-79分：中等AI味
  - 50分以下：高AI味（需优化）

**代码变更：**

| 文件 | 变更 |
|------|------|
| `src/app/components/CreateWorkbench.tsx` | 修改 `handlePolish` 和 `handlePolishFromContent` 函数，调整评分阈值显示 |
| `src/app/api/create-workshop/route.ts` | 修改 `polishContent` 函数支持针对性优化 |

3. **修改 CreateWorkbench.tsx**：

## v1.15.1 (2026-04-01)

### Bug修复 (PATCH) - 微信公众号草稿摘要长度限制

修复发布到微信公众号草稿箱时出现的 "description size out of limit" 错误。

#### 问题描述

发布文章到微信公众号草稿箱时报错：
```
⚠️ 创建草稿失败: description size out of limit hint: [v.K7ya007178-5] rid: 69cc8ccf-0ebb3b69-76a8ef12 (45004)
```

#### 原因分析

`/api/publish/route.ts` 中的 digest 生成逻辑有问题：
```javascript
// 错误的逻辑：先截取120字符，再去除HTML标签
const digest = content.slice(0, 120).replace(/<[^>]+>/g, '').replace(/\n/g, ' ') + '...';
```

这样处理后：
1. 先截取 120 字符（包含 HTML 标签）
2. 再去除 HTML 标签后，实际文本可能仍接近 120 字符
3. 再加上 `...` 就超出了微信公众号 API 的 120 字符限制

#### 修复方案

修改为先去除 HTML 标签，再截取到 117 字符：
```javascript
// 正确的逻辑：先去除HTML标签，再截取
const plainText = content.replace(/<[^>]+>/g, '').replace(/\n/g, ' ').replace(/\s+/g, ' ').trim();
const digest = plainText.length > 117 ? plainText.slice(0, 117) + '...' : plainText;
```

#### 修改文件

| 文件 | 变更 |
|------|------|
| `src/app/api/publish/route.ts` | 修复两处 digest 生成逻辑（第139行和第232行） |

## v1.14.0 (2026-04-01)

### 功能优化 (MINOR) - 润色优化系统重构

本次更新重构了润色优化系统，基于1.md完整规范进行润色，移除了冗余的去味清单检查步骤。

#### 1. 润色优化提示词升级

基于1.md的完整润色优化规范，包含：

**5条核心规则**：

- 删除填充短语 - 去除开场白和强调性拐杖词
- 打破公式结构 - 避免二元对比、戏剧性分段、修辞性设置
- 变化节奏 - 混合句子长度，两项优于三项
- 信任读者 - 直接陈述事实，跳过软化、辩解
- 删除金句 - 如果听起来像可引用的语句，重写它

**24种AI模式检测与修正**：

- 内容模式(1-6)：意义膨胀、名人提及、表面分析等
- 语言模式(7-12)：AI词汇、系词回避、负面平行等
- 风格模式(13-18)：破折号过度、粗体过度等
- 沟通模式(19-21)：聊天机器人痕迹、截止免责声明、谄媚语气
- 填充和模糊(22-24)：填充短语、过度模糊、通用结论

**领域适配**：

- 技术文档：仅删除填充词，保留术语
- 商务邮件：去除谄媚和填充短语
- 创意写作：应用叙事节奏，注入具体细节
- 营销文案：强调具体利益
- 学术写作：删除夸大和模糊归因

**个性与灵魂注入**：

- 有观点：对事实做出反应
- 变化节奏：短句与长句交替
- 承认复杂性：真实的人有复杂的感受
- 适当使用"我"：第一人称表明有真实的人在思考
- 允许一些混乱：完美的结构感觉像算法

#### 2. 去味清单移除

**原因**：去味清单（7项）与润色优化部分高度重合：

- "无模板词" → 对应"删除填充短语"和AI模式22
- "无完美三段式" → 对应"打破公式结构"和AI模式10
- "有个人经历" → 对应"个性与灵魂注入"
- "有情感词" → 对应"个性与灵魂注入"
- "有口语化表达" → 对应润色技巧
- "长短句交错" → 对应"变化节奏"
- "有括号补充" → 对应润色技巧

#### 2. Prompt 同步机制

**问题**: `initializePrompts()` 只会插入不存在的 prompt，不会更新已存在的 prompt，导致数据库中的 prompt 配置不会随代码更新。

**修复**: 修改 `initializePrompts()` 函数，使其能够同步更新已存在的 prompt 配置：

```typescript
// 修复前：只插入不存在的
if (!existingKeys.has(key)) {
  await db().insert(promptConfigs).values({...});
}

// 修复后：插入或更新
if (!existingKeys.has(key)) {
  await db().insert(promptConfigs).values({...});
} else {
  await db().update(promptConfigs)
    .set({ name, description, template, updatedAt: new Date() })
    .where(eq(promptConfigs.key, key));
}
```
