### 2.5 图片生成功能 ⚠️

**文件**: `src/app/api/image-generation/route.ts`, `src/lib/image/service.ts`

| 子功能 | 实现方式 | 实战可用 |
|--------|---------|---------|
| 单图生成 | 真实MiniMax API | ⚠️ 有Bug |
| 文章配图 | 真实MiniMax API + LLM生成prompt | ⚠️ 有Bug |
| 备选方案 | **无** | ❌ |

**🚨 发现严重Bug**：
- `getMiniMaxConfig()`（第46-62行）直接返回`minimaxConfig.apiKey`
- 但数据库中存储的API Key是**加密的**（AES-256-GCM格式：`iv:authTag:encrypted`）
- 代码**没有调用`decrypt()`解密**，直接把加密字符串当API Key发送给MiniMax
- 对比：`llm/service.ts`第111行正确调用了`decrypt(config.apiKey)`

**结果**：图片生成功能会因API Key无效而**必定失败**。

**其他问题**：
- 仅依赖MiniMax一个提供商，无备选方案
- MIME类型硬编码为`image/jpeg`，上传PNG可能导致格式不匹配

## 五、优先修复建议

1. **修复图片生成API Key解密Bug**
   - 在`getMiniMaxConfig()`中添加`decrypt()`调用
   - 同时修复`lib/image/service.ts`中的相同问题

3. **Cookie持久化**
   - 将Cookie存储到数据库或localStorage，而非useState

   - 调用已有的`getArticleStats`函数，添加定时同步机制

5. **数据分析去假数据**
   - 移除硬编码的`categoryStats`，改为基于文章实际标签/分类统计

### P2 - 中期规划（补齐功能缺口）

6. **实现通用爬虫真实逻辑**（或移除Mock入口避免误导）
7. **实现抖音/小红书热点抓取**

## 六、结论

这个项目的**核心创作+发布链路是真实可用的**（AI写作 → 排版 → 发布到微信草稿箱），但**上游数据获取层大量使用Mock数据**（热点搜索、通用爬虫、监控等），导致用户无法真正实现"自动发现热点 → 自动采集素材 → AI创作 → 自动发布"的完整闭环。

简单来说：**手动输入内容 → AI处理 → 发布微信** 这条路是通的；但 **自动获取热点/素材** 这条路基本不通。
